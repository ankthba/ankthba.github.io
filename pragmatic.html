How it works

How an agent lives in the kernel

Agents aren't processes or containers. On Pragmatic they're first-class kernel objects with persistent identity, cryptographic capabilities, and cognitive state the kernel tracks natively.

i

Bare-metal boot

A microkernel written entirely in Rust runs directly on ARM64 hardware. No Linux, no hypervisor, no garbage collector, and none of the abstraction tax that comes from bolting agent infrastructure onto an operating system designed for humans.

ii

Agents as kernel objects

Each agent is a first-class kernel object with its own persistent identity, cryptographic capabilities, and cognitive state. Not a process, not a container, but an entity the kernel understands and carries across sessions.

iii

Goal-aware execution

The scheduler allocates resources toward objectives, not just CPU cycles. It reasons about agent goals, progress, and priority, while every syscall and state transition is cryptographically signed and written to an immutable audit trail.

Architecture

Designed at the kernel level

Kernel

Bare-metal microkernel

Written entirely in Rust, running directly on ARM64 with no Linux, no hypervisor, and no garbage collector. The agent runtime is the operating system, not a layer sitting on top of one.

Security

Zero-trust capabilities

Every resource access requires an unforgeable capability token with cryptographic provenance. Agents run at strict least-privilege, enforced in hardware by the ARM64 MMU. There is no ambient authority, and agents physically cannot touch memory they don't own.

Scheduler

Goal-aware, not time-aware

Rather than treating CPU time as the primitive, the scheduler understands objectives, progress metrics, and priority hierarchies, allocating resources toward completing goals instead of merely running cycles.

Memory

Cognitive primitives

Working, episodic, semantic, and procedural memory are first-class kernel abstractions rather than external databases. Agents manage state across sessions natively, and every transition is signed into an immutable audit log.

Use cases

Where kernel-level guarantees matter

Financial services

Agents that trade and underwrite with kernel-enforced risk limits and cryptographically signed decisions. Every action is provable and every limit is enforced below the application layer, not promised by it.

Healthcare

HIPAA-compliant diagnostic agents with hardware-enforced data isolation. Patient data lives behind capability boundaries the MMU enforces, not behind policy an application asks you to trust.

Critical infrastructure

Autonomous control systems with hardware kill switches and least-privilege by construction. Agents operate physical systems without ambient authority over anything they don't explicitly own.

Enterprise AI

Multi-agent orchestration with full provenance tracking. Every syscall and state transition is signed and logged, so compliance is architectural rather than procedural.

Research

Agentical

A probabilistic process calculus for durable multi-agent LLM systems. Distributed-systems guarantees assume programs are deterministic; LLMs aren't. Agentical is the formal groundwork for reasoning about agent systems anyway, with replay, recovery, and information-flow guarantees that hold despite a nondeterministic model.

Preliminary research paper · Aniketh Bandlamudi · University of Virginia · programming-language theory and formal methods

The gap

Guarantees assume determinism

Replay, crash-recovery, debugging, and information-flow reasoning all assume programs are deterministic. Frontier models show only 50–96% determinism even at temperature 0, so LLM agent systems lose those guarantees. Prior formal work each gives up one of three needed properties.

i

LLMbda / FIDES

Precise single-agent calculi, but they assume the model is deterministic.

ii

MSC calculus

Multi-agent and Lean-mechanized, but it treats the LLM as a black box and cannot state quantitative or probabilistic guarantees.

iii

Durable execution (Temporal)

Crash-recovery in practice, but it requires determinism to work.

The idea

Model each LLM call as a first-class probability distribution, the Oracle, and journal every realized outcome to an append-only log, the Journal. Recovery becomes replay: read back recorded outcomes instead of re-sampling.

Two nondeterminisms are kept cleanly separate: probabilistic (the Oracle) and adversarial scheduling (resolved by a scheduler), the standard probabilistic-automata / MDP-with-schedulers setup.

The calculus

Six primitives, two modes

A π-calculus-style process calculus with a probabilistic small-step operational semantics in record and replay modes. A configuration is the 5-tuple <P;H;J;K;t>: agent pool, shared store, journal, channel state, and logical clock.

Three theorems

Stated with honest scope

T1 · Fully proved

Replay soundness

The flagship result: replaying the recorded journal reproduces the original run's trace exactly, with probability 1, proved via a coupling on the shared oracle-randomness space. Equiprobability follows.

T2 · Pen-and-paper

Probabilistic noninterference

Secret (High) inputs don't affect public (Low) observations, proved as termination-insensitive noninterference, conditional on named assumptions: per-label oracle partitioning and a label-filtered scheduler.

T3 · Pen-and-paper

Compositional bounds

Local per-agent probability contracts compose to a global bound, an assumption-free union bound. The product form holds only under independence, with a Fréchet–Hoeffding fallback otherwise.

Verified / built

From proof to running code

Lean 4 mechanization

lean/AgenticalT1.lean covers T1's operational core (the oracle fragment), trace-equivalence, crash-resume (C1), no-orphaned-effect (L4), and cursor-determinism (L1), all sorry-free. Axioms are propext, plus Quot.sound for L4. Not yet mechanized: channels, the measure-theoretic layer, or T2/T3.

Rust runtime

A zero-dependency reference implementation. Six integration tests pass demonstrating T1, C1, and L4. Measured: 1000/1000 replay-determinism; ~230 ns O(1) append latency, flat to 1M entries; and a ~1500× replay speedup under a 1 ms/draw latency-injected oracle, since replay performs zero oracle calls.

Novelty

The defensible claim is the intersection: probabilistic-operational semantics × multi-agent × durable-replay, backed by a reference implementation. No prior work occupies all three.

The closest, Semantic Fusion (arXiv:2601.12580), does probabilistic bisimulation, but over ontology-validated structured proposals rather than raw LLM output, and has no replay.
